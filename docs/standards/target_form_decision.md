**결정일**: 2026-04-23
**상태**: LOCKED — v1 기간 중 변경 금지

## 결정

**1차 대상**: **위험성평가표 본표 (위험성평가실시표)** — 1종 단독.

등록부·회의록·TBM은 **후속 단계로 분리**.

---

## 근거

### 1. 법적 지위
- 산업안전보건법 시행규칙 §37 에서 **기록·보존 의무**가 명시된 대상은 본표가 중심.
- 유해위험요인·위험성 결정·감소조치 3축이 본표 내에 모두 들어감.

### 2. KRAS 엔진과의 정합성
현재 이 프로젝트의 엔진 출력(`POST /api/v1/risk-assessment/build` + `table_builder.build_risk_table_from_result`)이 **본표의 "위험성평가실시" 시트 row 구조와 1:1 매핑 가능**한 형태로 이미 설계되어 있음.

| 엔진 결과 (현재) | 본표 (목표) | 매핑 가능성 |
|--------------|-----------|----------|
| work_type → process | 공정명 | ✓ 직접 |
| hazards[].hazard | 유해위험요인 | ✓ 직접 |
| hazards[].controls | 위험성 감소대책 | ✓ 직접 |
| hazards[].confidence_score | 위험성 (가능성×중대성) | △ 변환 규칙 필요 |
| hazards[].evidence_summary + references | 관련근거 | ✓ 요약 매핑 가능 |

반면 등록부(B)·회의록(C)·TBM(D)은 **본표 외의 운영 메타데이터**(회의참석자·일자·서명·일일현장기록)를 요구하며, 엔진이 자동 생성할 수 있는 성질의 정보가 아니다.

### 3. 나머지 3종의 종속성
- **등록부**: 본표를 집계한 상위 인덱스 → 본표 없이는 무의미
- **회의록**: 인적 참여 기록 → 수기 데이터 의존
- **TBM**: 일일 현장 기록 → 본표에서 row 발췌 + 일일 운영 데이터 결합 필요

따라서 본표를 먼저 확정하지 않으면 나머지 3종의 입력 소스가 모호해진다.

### 4. 출력 비용 대비 가치
- 본표 1종만 완성해도 **감독관 실사·원청 제출**의 70~80% 충족.
- 회의록·TBM은 현장 수기 기록을 표 양식에 얹는 단순 변환 작업 → 엔진 자동화 가치 낮음.

## v1 범위에 **포함하는 것**

1. 본표(위험성평가실시) 시트 1종 스키마 고정.
2. 본표의 **법적 필수 컬럼** 전체 지원:
   - 공정명, 유해위험요인, 현재안전보건조치(optional), 가능성, 중대성, 위험성, 감소대책.
3. 실무 권장 컬럼 중 **엔진이 자동 채울 수 있는 것**:
   - 관련근거(legal_basis), 개선후 위험성, 평가척도.
4. 상단 메타 영역(회사명/현장명/평가일자 등) — **입력 슬롯**으로만 정의 (값은 사용자 제공).

## v1 범위에 **포함하지 않는 것**

| 항목 | 사유 |
|------|------|
| 등록부 (B) | 본표 row 확정 후 집계 계층. v2에서 다룸. |
| 회의록 (C) | 수기 필드 위주. 자동화 가치 낮음. v2에서 fillable template 형태로. |
| TBM 일지 (D) | 일일 현장 수기. v3에서 본표 연계 기능과 함께. |
| 조직구성(시트2)·위험성기준표(시트3)·회의결과(시트4) | 회사/현장 정적 정보·채점 기준표. 템플릿 고정값으로만 취급, 엔진 자동 생성 대상 아님. |
| 엑셀(XLSX)·PDF 물리 출력 구현 | 이번 단계 **명시적 금지**. JSON 표준 구조만. |
| UI(화면) | 이번 단계 **명시적 금지**. |
