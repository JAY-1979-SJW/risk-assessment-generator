## 1. 확정 원칙

1. **법적 핵심 문서 우선** — 산안법 제36조·시행규칙 제37조의 "위험성평가실시표" 가 최우선.
2. **현재 엔진 결과와 자연스럽게 매핑 가능** — `form_builder` / `table_builder` 출력을 재가공 없이 흡수 가능한 레이아웃 우선.
3. **실제 원본 파일 확인 완료** 서식만 대상.
4. **1차 최소 범위**: 위험성평가 본표 **1종** + 작업계획서 **0종 (파일 부재로 후속 이관)**.
5. 등록부/회의록/TBM/점검표는 **v1 범위 외**로 명시 분리.

## 2. 1차 구현 대상 — 확정

### 대상 A: 위험성평가 본표 (위험성평가실시표)

| 항목 | 값 |
|------|----|
| 기준 원본 | `export/위험성평가표_공문기반_20250114_v2.xlsx` → 시트 `위험성평가실시` |
| 레퍼런스 원형 | `참조자료/위험성평가 표준안(경비).xls` → 시트 `위험성평가실시` (17컬럼 원형) |

| 컬럼 수 | **16** (A~P) |
| 헤더 행 | row 8 (셀 내 `\n` 으로 2단 표현) |
| 본문 시작 | row 9 |
| 상단 메타 | row 7 의 단일 행 (공정명 / 위험성평가 / 평가일시) |

| 이번 v1 구현 범위 | **이 1개 시트만** |

### 기준 레이아웃 선택 사유

1. **16컬럼 공문기반** 이 민원·공문 대응 실무에서 가장 광범위하게 사용되며, `docs/standards/kras_standard_form_v1.md` 가 공식 준거로 채택.
2. KRAS 17컬럼 원형과의 차이는 "비고" 1열뿐이므로, 추후 v1.2 에서 17컬럼으로 확장 시 하위 호환 유지 가능.
3. 현재 `form_builder` 스키마가 16컬럼과 가장 근접 (`hazard_category_major/minor`, `hazard`, `legal_basis`, `current_measures`, `risk_scale`, `probability`, `severity`, `risk_level`, `control_measures`, `residual_risk_level`, `target_date`, `completion_date`, `responsible_person`).

## 4. 1차 구현 대상 서식 — 레이아웃 요약

### 레이아웃 (공문기반 v2 `위험성평가실시` 시트 기준)

```
row 1 : 제목       [A1:P1 병합]   KRAS(표준 위험성평가)
row 2 : 부제       [A2:P2 병합]   (http://kras.kosha.or.kr)
row 3~6 : (공백)
row 7 : 상단 메타  [A7:C7]=공정명: …  [D7:G7]=위험성평가  [N7:P7]=평가일시: …
row 8 : 헤더 (16컬럼, 셀 내 개행으로 2단 표현. C8:E8 만 병합)
row 9~ : 본문 (hazard 1건 = 1행)
```

### 16컬럼 최종 확정 (공문기반 v2 실측)

| # | 컬럼 원문 (셀 개행 포함) | 병합 | form_builder 필드 |
|---|--------------------------|------|-------------------|
| 1 | 공정명 | — | `process` |
| 2 | 세부작업명 | — | `sub_work` |
| 3 | 위험분류 (대분류) | `C8:E8` (헤더만 병합) | `hazard_category_major` (현재 null) |
| 4 | (위험분류 중분류 — 데이터 row 에서만 분리) | 병합 소속 | `hazard_category_minor` (현재 null) |
| 5 | (유해위험요인 서술 — 데이터 row 에서만 분리) | 병합 소속 | `hazard` |
| 6 | 관련근거\n(법적기준) | — | `legal_basis` |
| 7 | 현재의\n안전보건조치 | — | `current_measures` (현재 null — 사용자 입력) |
| 8 | 평가\n척도 | — | `risk_scale` (고정 "3x3") |
| 9 | 가능성\n(빈도) | — | `probability` |
| 10 | 중대성\n(강도) | — | `severity` |
| 11 | 위험성 | — | `risk_level` |
| 12 | 위험성 감소대책 | — | `control_measures` (배열 → 줄바꿈 join) |
| 13 | 개선후\n위험성 | — | `residual_risk_level` |
| 14 | 개선\n예정일 | — | `target_date` |
| 15 | 완료일 | — | `completion_date` |
| 16 | 담당자 | — | `responsible_person` |

### 엔진 연결성 (직접 매핑 가능성)

- **자동 채움 가능 필드** (엔진 산출): 1, 2, 5, 6, 8, 9, 10, 11, 12, 13.
- **사용자 입력 필드**: 7 (현재조치), 14 (개선예정일), 15 (완료일), 16 (담당자).
- **v1 에서 항상 공란 유지**: 3, 4 (위험분류 대/중 — 엔진이 분류 메타 아직 없음).

세부 스키마는 `docs/standards/form_schema_lock.md` (5단계) 에서 확정.

## 6. 주의 — 현행 Excel 렌더러와의 정합 검토

현재 구현된 `engine/output/form_excel_builder.py` / `docs/standards/excel_layout_spec.md` 는 **17컬럼**(공문기반 16컬럼에 "세부작업명"이 별도 배치되어 담당자까지) 레이아웃을 사용 중.

직접 확인 결과 공문기반 v2 원본은 **16컬럼**이다. 따라서:

- 현행 렌더러의 17컬럼은 원본의 16컬럼에서 "세부작업명"을 3번 슬롯에 별도 배치한 **파생 변형**.
- **v1 스키마 잠금 단계(5단계)** 에서는 **원본 16컬럼** 을 기준으로 스키마를 확정하고, 렌더러를 재정렬하는 작업을 **별도 후속 태스크**로 분리한다.
- 이번 단계의 스키마 잠금 결과와 현재 렌더러의 정합 여부는 5단계에서 명시한다.
