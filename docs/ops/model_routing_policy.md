# 모델 선택 운영 기준

## 1. 목적

안전서류 자동생성기 개발에서 비용을 줄이면서도 오류로 인한 재작업을 최소화하기 위한 명확한 모델 선택 기준을 수립한다.

### 역할 분담 원칙
- **GPT (최상위 모델)**: 감리·설계·지시문 작성 역할
- **Claude 중간모델**: 실행·문제 해결·검증 역할
- **Claude 하위모델**: 정해진 패턴 반복 실행 역할

**핵심 원칙**: 모델 성능보다 작업 분리와 검증 단계가 비용 효율성을 결정한다.

---

## 2. 기본 원칙

### 2.1 하위 모델 사용 원칙
- 정해진 패턴을 반복하는 작업에만 사용
- 허용된 파일 범위가 명확한 작업만 선택
- 예: 기존 builder 복제, catalog YAML 수정, smoke test 추가

### 2.2 중간모델 사용 원칙
- 실패 원인 분석 필요 시
- 서버/스크립트/배포 작업에 필수
- 5개 이상 파일을 동시에 수정하는 작업
- 작업 범위 판단이 필요한 경우

### 2.3 서버·배포·스크립트 작업 원칙 (중간모델 이상 필수)
- shell quoting, SSH, docker, git script 수정
- 배포 자동화 스크립트 변경
- bash -c, heredoc, remote command 문제
- git history 꼬임, rebase/reset 판단 필요

### 2.4 검증 단계 분리 원칙
- push/server 반영은 반드시 별도 단계로 분리
- 각 단계 종료 후 상태 확인 및 보고
- 다음 단계 시작 전 사용자 승인 대기 가능

### 2.5 공통 원칙 (모든 모델)
- 허용 파일 외 수정 금지
- 완료 후 명시적 STOP
- 명확하지 않은 범위는 수행하지 않음

## 3. 하위 모델 사용 가능 작업

| 작업 분류 | 구체적 작업 | 조건 |
|---------|-----------|-----|
| **읽기 전용 조사** | read-only catalog 조사 | 수정 없음 |
| | grep/find 기반 상태 확인 | 읽기만 수행 |
| **기존 패턴 복제** | 기존 builder 패턴 복제 | 유사한 builder 코드 존재 |
| | form_registry에 FormSpec 1개 추가 | 템플릿 명확 |
| | document_catalog.yml에서 서식 DONE 전환 | YAML 수정만 |
| **자동화 작업** | smoke_test_p1_forms.py에 smoke test 추가 | 기존 테스트 구조 복제 |
| | py_compile 실행 | 컴파일만 수행 |
| | validate_form_registry 실행 | 검증 스크립트 실행 |
| | lint_safety_naming 실행 | 린트 스크립트 실행 |
| | smoke_test_p1_forms 실행 | 테스트 실행 |
| **상태 확인** | git status 확인 | 읽기만 수행 |
| | 서버 상태 read-only 확인 | ps, ls, tail 등 읽기 명령 |
| **커밋 및 배포** | 단일 목적 커밋 생성 | 기존 커밋 메시지 패턴 따름 |
| | push만 수행 (git pull → push 없음) | 배포 스크립트 미수정 |
| **문서 작성** | 보고서 Markdown 파일 1개 생성 | 코드/구조 변경 없음 |
| | dry-run 결과 보고서 작성 | 읽기 기반 정리 |

### 3.1 하위 모델 작업 체크리스트
- ✓ 허용 파일이 명확한가?
- ✓ 기존 유사 패턴이 있는가?
- ✓ 실패해도 자동 수정하지 않을 수 있는가?
- ✓ 단계 종료 후 STOP할 것인가?
- 모두 ✓이면 하위 모델 배정 가능

## 4. 중간모델 이상 사용 권장 작업

| 작업 분류 | 구체적 작업 | 이유 |
|---------|-----------|-----|
| **설계·분석** | 신규 서식 구조 설계 | 법령 해석 필요 |
| | 법령 근거 해석 또는 서식 필드 설계 | 법률 판단 필요 |
| | 기존 패턴이 없는 builder 구현 | 새로운 로직 설계 |

| | registry 전체 구조 변경 | 의존성 분석 필요 |
| | 공통 Excel 렌더러 수정 | 모든 서식 영향 |
| **서버/배포 작업** | 배포 스크립트 수정 | SSH key, bash 문제 위험 |
| | SSH key 처리 | 보안 위험 |
| | Windows Git Bash quoting 문제 | 환경 특화 문제 |
| | bash -c / heredoc / remote command 문제 | 복잡한 쉘 명령 |
| | docker compose 경로 문제 | 컨테이너 경로 복잡도 |
| **문제 해결** | 서버 git pull 실패 원인 분석 | 원인 진단 필요 |
| | git history 꼬임, rebase/reset 판단 | 복원력 필요 |
| **재시도 작업** | 같은 단계에서 1회 이상 실패한 작업 | 원인 분석 필수 |
| **대규모 변경** | 5개 이상 파일을 동시에 수정하는 작업 | 의존성 관리 필요 |
| **데이터 작업** | 데이터 수집 스크립트 변경 | 복잡도 높음 |
| | 법령/evidence 수집 설계 | 정책 판단 필요 |

## 5. 최상위 모델(GPT)이 담당해야 할 작업

### 5.1 전략·감리
- 전체 우선순위 결정
- 다음 구현 문서 선정
- 모델 전환 여부 판단

### 5.2 정책·해석
- 법령/실무 문서 체계 판단
- 상품화 방향 판단
- 서식 카테고리 구조 재설계

### 5.3 위험 관리
- 실패 보고 해석 및 조치 방향
- 지시문 재작성
- 서버 작업 승인 여부 판단
- 큰 리팩토링 여부 판단

## 6. 모델 전환 기준

### 6.1 하위 모델 → 중간모델 전환 조건 (즉시 전환)
- ⚠️ **같은 오류 1회 이상 반복**
- ⚠️ **push/commit 단계를 혼동**
- ⚠️ **서버 경로/compose 경로를 혼동**
- ⚠️ **검증 기준을 잘못 잡음**
- ⚠️ **shell quoting 문제 발생**
- ⚠️ **SSH 인증/키 문제 발생**
- ⚠️ **docker compose 실패 발생**
- ⚠️ **예상 외 파일 수정 발생**
- ⚠️ **지시문 범위를 벗어난 작업 수행**
- ⚠️ **작업 보고가 앞 단계 보고를 반복**

### 6.2 중간모델 → GPT 검토 전환 조건
- 법령 해석이 필요한 경우
- 카테고리 구조를 바꿔야 하는 경우
- 전체 catalog 구조 변경이 필요한 경우
- 공통 스키마 변경이 필요한 경우
- 운영 정책 변경이 필요한 경우

## 7. 단계별 작업 표준

### 7.1 서식 구현 표준 흐름 (CL-XXX 구현 작업)

| 단계 | 작업 | 설명 |
|-----|-----|------|
| 1 | read-only 조사 | 기존 법령·서식 조사 및 구조 분석 |
| 2 | 구현 설계 | FormSpec 필드 설계 (GPT 또는 중간모델) |
| 3 | builder 파일 생성 | builder_{cl_code}.py 작성 |
| 4 | builder 안전성 확인 | py_compile, import 테스트 |
| 5 | form_registry 등록 | form_registry.py에 FormSpec 추가 |
| 6 | document_catalog DONE 전환 | document_catalog.yml 상태 변경 |
| 7 | smoke test 추가 | smoke_test_p1_forms.py에 테스트 추가 |
| 8 | 전체 검증 | validate_form_registry, lint_safety_naming 실행 |
| 9 | 기능 커밋 | git commit (코드 변경만) |
| 10 | push | git push origin master |
| 11 | 서버 dry-run | ssh에서 `git pull --ff-only` dry-run |
| 12 | 서버 git pull + health | `git pull --ff-only`, app health check |
| 13 | restart + health | docker restart, endpoint 재검증 |
| 14 | 최종 요약 | 결과 보고 (PASS/WARN/FAIL) |

### 7.2 각 단계별 STOP 포인트
- **단계 2 완료 후**: 설계 승인 대기
- **단계 8 완료 후**: 모든 검증 PASS 확인
- **단계 10 완료 후**: push 결과 확인
- **단계 13 완료 후**: 서버 반영 완료

## 9. 이번 사례에서 얻은 교훈 (CL-008, CL-009)

### 9.1 성공 사례
- **CL-008/CL-009 builder 구현**: 하위 모델만으로 성공
  - 기존 builder 패턴이 명확했음
  - FormSpec 필드 구조가 단순했음
  - 허용 파일 범위가 명확했음

### 9.2 문제 발생 사례

**FormSpec document_id 검증 오류**
- 문제: 하위 모델이 document_id 유효성을 놓침
- 원인: FormSpec 구조 이해 부족
- 해결: GPT 검수 필요
- 교훈: FormSpec 설계 단계에서 GPT 검수 필수

**서버 배포 스크립트 문제**
- 문제: SSH key, bash -c, git -C, docker compose -f 오류 반복
- 원인: 하위 모델의 shell/SSH/배포 지식 부족
- 재시도: 3회 이상 반복, 오래 걸림
- 교훈: 첫 시도부터 중간모델 사용했다면 1-2시간 절약 가능

### 9.3 비용 분석
| 시나리오 | 모델 | 예상 시간 | 실제 시간 | 교훈 |
|--------|-----|---------|---------|------|
| builder 구현 (패턴 있음) | 하위 | 1-2h | 1-2h | ✓ 정확 |
| FormSpec 설계 (신규) | 하위 | 2h | 4-5h | ✗ GPT 필요 |
| 배포 스크립트 수정 | 하위 | 1-2h | 4-6h | ✗ 중간모델 필수 |

## 10. 최종 운영 규칙

### 10.1 모델의 역할 정의
```
"하위 모델은 실행자, 중간모델은 문제 해결자, GPT는 감리자다."
```

- **하위 모델**: 정해진 패턴을 따라 실행
- **중간모델**: 오류 원인을 분석하고 해결
- **GPT**: 전체 흐름을 감리하고 정책 결정

### 10.2 비용 효율성 원칙
```
"하위 모델은 싸지만, 실패가 반복되면 총비용과 시간이 증가한다."
```

- 첫 시도가 중요: 오류 원인이 불명확하면 즉시 상위 모델로 전환
- 같은 오류 2회 = 상위 모델로 전환 신호
- 총 비용 = (모델 비용 × 횟수) + (재작업 시간)

### 10.3 서버·배포·스크립트 작업 원칙
```
"서버·배포·스크립트 문제는 처음부터 중간모델을 사용한다."
```

- SSH, bash -c, heredoc, docker compose: 중간모델 이상
- git history 분석, rebase/reset 판단: 중간모델 이상
- 배포 자동화 스크립트 수정: 중간모델 이상
- 예외: read-only 상태 확인만 하위 모델 가능

### 10.4 단계별 작업 관리 원칙
```
"작업은 항상 작은 단계로 나누고, 각 단계는 PASS/WARN/FAIL로 종료한다."
```

| 결과 | 의미 | 다음 액션 |
|-----|-----|---------|
| **PASS** | 예상된 결과 달성 | 다음 단계 진행 |
| **WARN** | 경고 또는 예상 외 동작 | 사용자 확인, 원인 분석 |
| **FAIL** | 작업 실패 또는 오류 | 원인 분석, 상위 모델 전환 검토 |

### 10.5 명시적 완료 신호
- 모든 작업은 명시적 **STOP** 으로 종료
- STOP 전에 변경 상태 확인: `git status --short`
- STOP 전에 핵심 검증 완료: 해당 테스트 패스
- 사용자 재입력 전까지 자동 재시작 금지

## 11. 의사결정 플로우

```
작업 요청
    ↓
[단계 파악] → 서식 구현 / 배포 / 설계 / 분석
    ↓
[범위 확인]
    ├─ 명확한 패턴 있음? → YES → 하위 모델
    ├─ 서버/배포/스크립트? → YES → 중간모델
    ├─ 법령/카테고리/정책? → YES → GPT
    ├─ 이전에 실패함? → YES → 중간모델 이상
    └─ 5개+ 파일 동시 수정? → YES → 중간모델 이상
    ↓
[모델 배정 및 작업 수행]
    ↓
[단계별 검증] → PASS/WARN/FAIL
    ↓
[완료 또는 재배정]
```

---

**최종 원칙**: 비용 절감과 품질은 상충 관계가 아니다. 올바른 모델을 올바른 단계에 배정할 때, 비용과 품질 모두 최적화된다.
