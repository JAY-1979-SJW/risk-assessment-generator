# 문장 품질 저하 유형 분류 (v0.1)

> 목적: 위험성평가 엔진 입력 품질을 떨어뜨리는 **원문 문장 수준의 품질 이슈**를
> 12개 유형으로 체계화한다. 이후 단계의 정제(split/normalize/noise/subject 보정) 규칙은
> 본 taxonomy 의 유형 코드를 기준으로 작성한다.
>
> 본 문서는 "문장을 예쁘게 고치는" 기준이 아니라,
> "엔진이 판단할 수 있는 최소 단위로 분해·구조화"하기 위한 품질 이슈 사전이다.
>
> 적용 대상: `sentence_labeling_sample_v2.csv` (800 샘플, kosha 520 / law 120 / expc 60 / admrul 60 / licbyl 40).
> 산출 연계: `sentence_normalization_schema.md`, `sentence_normalization_rules.md`,
>            `sentence_normalization_sample_v1.csv`.

---

## 0. 용어 정의

| 용어 | 설명 |
|------|------|
| rule_core | 실제 의무·금지·조치의 핵심이 되는 문장 단위 |
| candidate | 확정이 아닌 후보값. reviewer 검토 전 자동 채움 |
| split | 한 원문 문장을 의미 단위로 분해하는 작업 |
| normalize | 추상 표현/노이즈/주어 결손을 보정해 판단 가능한 형태로 바꾸는 작업 |
| noise | 의무·조치 판단에 기여하지 않는 설명·배경·정의·참조 문장 |
| noise 복구 | noise 로 분류된 문장 중 제어 신호가 결합된 것을 rule 로 되살리는 판정 |

## 1. 품질 저하 유형 12종 개요

| 코드 | 이름 | 한줄 요약 | 자동화 난이도 |
|------|------|-----------|----------------|
| Q01 | multi_action_sentence | 한 문장에 2개 이상 의무/조치/행위가 혼합 | 중 (패턴 split 가능) |
| Q02 | omitted_subject | 주어(행위 주체)가 생략됨 | 상 (문맥 의존) |
| Q03 | omitted_object | 동작의 대상(무엇을)이 생략됨 | 상 (문맥 의존) |
| Q04 | vague_expression | "적절히", "필요한", "충분히" 등 추상 표현 | 하 (사전 기반) |
| Q05 | descriptive_noise | 설명·배경·정의·홍보성 비의무 문장 | 중 (키워드 + 반례) |
| Q06 | mixed_condition_rule | 조건문과 의무문이 한 문장에 혼합 | 중 (접속사 분리) |
| Q07 | mixed_control_hazard | 위험요인 설명과 대책이 같은 문장에 혼합 | 중 (신호어 조합) |
| Q08 | evidence_only | 법조문/참조/근거만 있고 직접 대책 없음 | 하 (조문 패턴) |
| Q09 | msds_formulaic_text | MSDS 상투문구/일반 경고문 | 하 (사전 기반) |
| Q10 | duplicate_semantics | 표현만 다르고 같은 의미 반복 | 상 (의미 유사) |
| Q11 | incomplete_sentence | 절/항 일부만 있어 독립 판단 불가 | 중 (길이/종결어미) |
| Q12 | context_required | 인접 문맥 없이는 해석 불가 | 상 (문맥 의존) |

## 2. 유형별 상세

### Q01. multi_action_sentence — 복합 행위 혼합

- **정의**: 하나의 문장 안에 두 개 이상의 의무·조치·행위 동사가 결합되어,
  엔진이 단일 control/action 후보로 매핑하기 어려운 경우.
- **판별 기준**:
  - "A하고 B하여야 한다", "A·B·C" 열거, "및/또는/그리고" 결합 후 동사 종결
  - 동사 종결어미(`~하여야 한다`, `~한다`, `~할 것`)가 2회 이상 등장
  - 열거 기호(`·`, `·`, `,`, `①②③`) 뒤 동사구 반복
- **예시**:
  - "사업주는 국소배기장치를 설치하고 정기적으로 점검하여야 한다."
  - "안전모·안전대·보안경을 지급·착용시키고 교육하여야 한다."
  - "작업계획서를 작성·게시·비치하여야 한다."
- **엔진상 문제점**:
  - action_type 후보 하나로 고정 불가 → 정밀도 저하
  - control 후보 1:N 매핑을 놓치면 점검/교육/설치 등 별도 control 이 소거됨
- **정제 전략**: Q01 은 **split 의 1차 대상**. `split_group_id` 로 묶고 각 행위를
  `split_order` 로 나눠 subject/object/legal_ref 를 상속한다.
- **자동 처리 가능 여부**: 가능(중). 단, 동사 상속 규칙 필요.
- **사람 검토 필요 여부**: 애매한 의미 결합("설치·운영", "관리·감독")은 reviewer 검토.

### Q02. omitted_subject — 주어 생략

- **정의**: "설치하여야 한다", "점검하여야 한다"처럼 주어(사업주/근로자/도급인 등)가
  문장 자체에 표기되지 않은 경우.
- **판별 기준**:
  - 문장 시작 어절이 체언+조사(`~는/~이/~가/~은`) 형태의 주격이 아님
  - 동사 종결어미(`~하여야 한다`, `~한다`)는 있으나 주체 명사구 없음
  - 법령 본문에서 자주 발생(항 번호 다음 바로 동사 시작)
- **예시**:
  - "매월 지급하여야 한다." (앞 조문 주어 "도급인" 상속 필요)
  - "정기적으로 점검하여야 한다."
- **엔진상 문제점**:
  - subject_type 후보 비움 → `mixed_or_unknown` 비율 증가
  - 규정 적용 대상(사업주 의무 vs 근로자 의무)이 뒤섞임
- **정제 전략**:
  - 같은 `document_id` + 직전 문장에서 주어가 확정적이면 `subject_candidate` 상속
  - 확정 불가 시 공란 유지, `note: subject_inherit_uncertain`
  - **임의 확정 금지** (원칙 고정)
- **자동 처리 가능 여부**: 부분(문맥 탐색 필요). 보수적 복원.
- **사람 검토 필요 여부**: 필요. 특히 조문 주어 이동 구간.

### Q03. omitted_object — 대상(목적어) 생략

- **정의**: "설치하여야 한다"에서 무엇을 설치하는지, "교육하여야 한다"에서
  무엇에 대해 교육하는지가 생략된 경우.
- **판별 기준**:
  - 동사 앞 `~을/~를` 목적어 절 없음
  - 같은 조문·같은 항목에서 선행 명사구(장비/문서/행위)가 있을 때 복원 시도
- **예시**:
  - "지정하고 이를 고용노동부장관에게 신고하여야 한다." (무엇을 지정?)
  - "작성하여 근로자에게 주지시켜야 한다." (무엇을 작성?)
- **엔진상 문제점**:
  - control_candidate / equipment_candidate 매핑 실패
  - action_type 만 잡히고 대상이 비어서 control 집계가 허술해짐
- **정제 전략**:
  - 같은 문장 앞 명사구 후보(관리자/장비/문서) → `object_candidate` 로 복원
  - **object 복원이 subject 복원보다 엔진 기여도가 크다 → 우선 복원 대상**
- **자동 처리 가능 여부**: 부분. 사전 기반 +조문 선행 명사구 탐색.
- **사람 검토 필요 여부**: 필요(문맥 상속).

### Q04. vague_expression — 추상 표현

- **정의**: "적절히", "필요한", "충분히", "즉시", "신속히", "안전하게",
  "확실히", "철저히", "이상이 없도록" 같이 구체성이 없는 수식어.
- **판별 기준**: 사전(아래 고정 어휘) 포함 여부.
  - `적절히`, `적절한`, `적절하게`
  - `필요한`, `필요시`, `필요에 따라`
  - `충분히`, `충분한`
  - `즉시`, `신속히`, `지체 없이`
  - `안전하게`, `확실히`, `철저히`
  - `이상이 없도록`
- **예시**:
  - "적절한 보호구를 착용하여야 한다."
  - "필요한 조치를 하여야 한다."
  - "즉시 대피시켜야 한다."
- **엔진상 문제점**:
  - 제어 강도·주기·수량 기준이 없어 판단 엔진에서 무게가 붕 뜸
  - control 후보로 써도 "적절한 조치" 같은 공허 문장만 양산
- **정제 전략**:
  - 삭제 가능형 ("적절한 보호구" → "보호구")은 수식어 삭제 + `note: 적절한`
  - 의미 손실형 ("필요한 조치") 은 원형 유지하되 `ambiguity_flag=Y`
  - control 후보 품질이 low 로 빠지면 control 매핑에서 lowweight
- **자동 처리 가능 여부**: 가능(사전 기반).
- **사람 검토 필요 여부**: 경미한 케이스는 자동, 의미 손실 가능성 시 검토.

### Q05. descriptive_noise — 설명/배경/홍보 문장

- **정의**: 의무·금지·조치 신호가 결합되지 않은 순수 설명·배경·정의·홍보 문장.
- **판별 기준**:
  - `requirement/prohibition/caution` 신호 전무
  - `참고`, `정보`, `특성`, `설명`, `안내`, `개요`, `배경`, `증상`, `정의` 등 단독 어휘
  - 숫자 임계값(톤/미터/명) 없음
- **예시**:
  - "해당 물질은 황색 액체이다."
  - "자세한 내용은 MSDS 를 참고하세요."
  - "산업안전은 매우 중요하다."
- **엔진상 문제점**:
  - control 후보 집계를 오염시켜 정밀도 저하
  - 현장 대책 문서에 그대로 들어가면 공허한 안내문이 됨
- **정제 전략**:
  - `sentence_role = explanation` + `noise_flag = Y`
  - 하지만 **복구 신호(보호구/환기/점검주기/허가/감시인/출입통제 등)가 결합**되면
    Q05 에서 탈락시키고 해당 rule 타입으로 복귀(`controls_noise_and_rendering.md` 1.3 참조).
- **자동 처리 가능 여부**: 가능(중). 반례(복구 신호) 체크 필수.
- **사람 검토 필요 여부**: 복구 후보만 검토.

### Q06. mixed_condition_rule — 조건+의무 혼합

- **정의**: 같은 문장에 조건절과 의무문이 결합되어 있어,
  조건이 엔진의 적용 조건에 전달되기 어려운 경우.
- **판별 기준**:
  - `~인 경우`, `~할 때`, `~시`, `~이상`, `~때에는`, `~를 초과하면`
  - 조건절 이후 `하여야 한다` / `한다` / `금지한다`
- **예시**:
  - "높이 2m 이상에서 작업하는 경우 안전대를 착용하여야 한다."
  - "밀폐공간 진입 전 산소농도를 측정하고 기준값을 초과하면 진입을 금지한다."
- **엔진상 문제점**:
  - condition_trigger 와 rule 이 섞여 sentence_type 1개로만 라벨되면 한쪽 손실
- **정제 전략**:
  - 조건절과 rule 절을 split 하여 `sentence_role` 을 각각 `condition`, `rule_core` 로 분리
  - condition 은 `condition_candidate`, rule 은 `control_candidate` 로 매핑
  - 조건은 rule 문장 쪽에 `condition_candidate` 로 상속
- **자동 처리 가능 여부**: 가능(중). 접속사 구분 규칙 필요.
- **사람 검토 필요 여부**: 다단 조건("A이면서 B일 때") 은 reviewer.

### Q07. mixed_control_hazard — 위험+대책 혼합

- **정의**: 위험요인 서술과 대책이 한 문장에 함께 있어,
  대책만 떼어 control 에 매핑하기 어려운 경우.
- **판별 기준**:
  - 위험 어휘(추락, 낙하, 감전, 질식, 끼임, 화재 등) + 조치 동사(설치/지급/착용/측정 등)
- **예시**:
  - "추락 위험이 있는 작업장에는 안전난간을 설치하여야 한다."
  - "감전 위험이 있는 충전전로 근접작업 시 절연용 보호구를 착용하여야 한다."
- **엔진상 문제점**:
  - hazard_candidate 와 control_candidate 가 같은 문장에서 경쟁 → 한쪽 소거
- **정제 전략**:
  - split 분리가 가능하면 hazard_statement + control_statement 로 분해
  - split 불가형("추락 위험 작업장에 …를 설치") 은 한 문장 유지 +
    `hazard_candidate` 와 `control_candidate` 를 동시에 채움
- **자동 처리 가능 여부**: 부분. 분리 가능 여부 판단 필요.
- **사람 검토 필요 여부**: 분리 불가형은 reviewer.

### Q08. evidence_only — 근거/참조만 있음

- **정의**: 법조문 번호, 별표 번호, 고시 참조만 있고 실체 의무가 없는 문장.
- **판별 기준**:
  - `제○○조`, `제○항`, `별표`, `고시 제`, `같은 법`, `이 규칙`
  - 해당 토큰 외 실체 의무 동사 없음
- **예시**:
  - "산업안전보건법 제29조에 따라 실시한다." (실체 없이 참조만)
  - "별표 4 에 따른다."
- **엔진상 문제점**:
  - control 매핑 불가, 근거만 남아 있어 대책으로 못 씀
- **정제 전략**:
  - `sentence_role = evidence` + `control_candidate` 비움
  - 인접 rule_core 문장의 `evidence_candidate` 로 상속 가능하면 노트 남김
- **자동 처리 가능 여부**: 가능(하).
- **사람 검토 필요 여부**: 불필요(특이 케이스 제외).

### Q09. msds_formulaic_text — MSDS 상투문구

- **정의**: MSDS/SDS 표준 템플릿에서 나오는 일반 설명·경고 상투문구.
- **판별 기준**:
  - "본 물질은", "증상이 나타날 수 있습니다", "자세한 내용은 MSDS 참고",
    "응급처치 정보는 ○○ 참조", "경고표지를 확인하세요"
- **예시**:
  - "피부에 접촉하면 증상이 나타날 수 있습니다."
  - "본 물질의 특성은 물리화학적 성질에 따른다."
- **엔진상 문제점**:
  - 실제 제어에 쓸 수 없음. descriptive_noise 의 하위 유형이지만 빈도가 높아 별도 유형화.
- **정제 전략**:
  - Q05 와 동일한 noise 처리 + `normalization_note: msds_formulaic`
  - 단, "경고표지 부착·게시" 같이 의무 동사 결합 시 `document_rule` 로 복귀.
- **자동 처리 가능 여부**: 가능(하).
- **사람 검토 필요 여부**: 복구 후보만 검토.

### Q10. duplicate_semantics — 의미 중복

- **정의**: 표현만 다르고 같은 의미를 반복하는 문장.
- **판별 기준**:
  - 같은 document_id 내에서 유사한 동사·목적어·주어 결합의 문장 반복
  - (완전 자동 판별은 어려움 → 본 단계에서는 후보 플래그만 생성)
- **예시**:
  - "방호장치를 설치하여야 한다." / "회전부에 방호 덮개를 갖추어야 한다."
- **엔진상 문제점**:
  - control 후보 count 왜곡, reviewer 수작업 가중
- **정제 전략**:
  - 본 단계에서는 **탐지만** 수행하고 자동 통합 금지
  - 같은 `document_id` 내 `action_candidate + object_candidate` 동일 시 후보 플래그
- **자동 처리 가능 여부**: 부분(정확도 낮음).
- **사람 검토 필요 여부**: 필요.

### Q11. incomplete_sentence — 절/항 단편

- **정의**: 법령 수집 과정에서 조문 번호/항 일부만 잘려 단독 문장으로 유입된 경우.
- **판별 기준**:
  - 문장 길이 < 8 어절 + 종결어미 없음
  - "제○○조(…)" 헤더 직후 짧은 토큰
  - "나.", "다.", "3." 같은 항 번호 단독
- **예시**:
  - "사업장의 명칭 및 소재지 3." (항 라벨 잔존)
  - "제5조(고용관리 책임자) ① ①"
- **엔진상 문제점**:
  - 라벨링 자체가 무의미, sentence_type 가 `condition_trigger/definition` 로 오분류되기 쉬움
- **정제 전략**:
  - 항 번호·장절 헤더는 `sentence_role = metadata` 로 분리
  - 본 단계에서 본문과 재결합은 하지 않음(수집 단계 이슈로 이관)
- **자동 처리 가능 여부**: 가능(하). 길이/패턴 기반.
- **사람 검토 필요 여부**: 불필요.

### Q12. context_required — 문맥 필수

- **정의**: 인접 문맥(앞뒤 조문, 같은 항목)의 정보 없이는 해석 불가한 문장.
- **판별 기준**:
  - 지시어(`이 경우`, `그 밖에`, `같은 조`, `해당`) 포함
  - subject_candidate 비고 object_candidate 비고 조건 모두 결손
- **예시**:
  - "이 경우 임금비용은 매월 지급하여야 한다." ("이 경우"의 선행 조건 필요)
  - "그 밖에 안전조치를 하여야 한다." (무엇의 밖?)
- **엔진상 문제점**:
  - control_candidate 를 자동으로 채울 수 없음
- **정제 전략**:
  - `context_required_flag = Y`
  - `subject/object/condition` 후보 비움 또는 상속 시도
  - control 후보 매핑은 보수적으로(저 confidence)
- **자동 처리 가능 여부**: 부분. 지시어 사전 기반 flag만.
- **사람 검토 필요 여부**: 필요.

## 3. 유형 간 우선순위 (동시 적용 시)

한 문장에 여러 유형이 동시에 해당할 수 있다. 적용 순서는 다음과 같다.

1. Q11 (incomplete_sentence) → 사전 탈락(rule 대상 제외, metadata 로 분리)
2. Q08 (evidence_only) → rule 으로 쓰지 않음
3. Q05/Q09 (noise) → noise 분기 후 복구 시도
4. Q01 (multi_action) → split 수행
5. Q06 (mixed_condition) → condition/rule 분해
6. Q07 (mixed_control_hazard) → 분리 가능 시 분해, 불가 시 결합 유지
7. Q02/Q03 (omitted_subject/object) → candidate 복원 시도
8. Q04 (vague_expression) → 정규화 + flag
9. Q12 (context_required) → flag 부여
10. Q10 (duplicate_semantics) → 탐지 flag 만(통합 금지)

## 5. 적용 원칙 (재확인)

- 원문(`raw_sentence_text`)은 삭제하지 않는다.
- 정제 결과는 별도 레코드(`normalized_sentence_text`)로 저장하며 원문을 덮어쓰지 않는다.
- subject/object 복원은 candidate 상태로만 남기고 자동 확정하지 않는다.
- noise 는 자동으로 운영 DB 반영하지 않는다.
- 품질 지표를 만들기 위해 과도한 split/삭제를 하지 않는다.
